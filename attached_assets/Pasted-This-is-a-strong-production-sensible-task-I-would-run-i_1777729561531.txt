## What I would tighten before running

### 1. Advanced configuration behaviour needs to be explicit

The task says the CreateWizard modal remains accessible via “Advanced configuration”. But the new page is separate from `DiagnosticsPage`, where that modal currently lives. A link alone cannot open a modal on another page unless you implement a query-param or route state.

Refine this:

```txt
Advanced configuration should navigate back to the existing DiagnosticsPage with a query param, e.g. /diagnostics?advancedCreate=1 or /client/:slug/diagnostics?advancedCreate=1, and DiagnosticsPage should open the existing CreateWizard when that param is present.

If query-param modal opening is too much for this task, the Advanced configuration link may route to the DiagnosticsPage with existing create button visible, but it must not be a dead link.
```

That prevents ambiguity.

### 2. “All six scenarios” is inaccurate

Question 1 has three options and Question 2 has four options, so there are **12 combinations**, not six. Your written recommendation rules group them, which is fine. Change wording to:

```txt
correct recommendation logic across the specified grouped cases
```

or explicitly test the important cases only.

### 3. Route placement in `App.tsx` must be precise

Because the app has WDC layout and ClientLayout, you want:

* `/create-insight` inside WDCLayout authenticated routes
* `/client/:slug/create-insight` inside ClientLayout client routes

Do not add it to public route blocks.

Add:

```txt
The page should render inside the same layout context as the diagnostics page: WDCLayout for /create-insight and ClientLayout for /client/:slug/create-insight.
```

### 4. Fast Diagnostic CTA should not say “Open Scorecard builder” if no route is available

The route `/client/:slug/config` exists, but whether that is truly the scorecard builder depends on client setup. Add:

```txt
If /client/:slug/config is not appropriate for the current client or no scorecard exists, show explanatory copy rather than forcing navigation.
```

This may be hard to detect now, so for v1 route to config and keep the copy cautious.

### 5. “No engine/schema imports” test may be brittle

Testing no engine imports from a component is unusual. Fine if project already has a pattern, but otherwise make it a code review requirement rather than a test.

```txt
No engine/schema imports in CreateInsightPage. Verify by grep/code review rather than a brittle unit test if needed.
```
